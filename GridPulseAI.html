📋 Non-Technical Overview

Context, Problem & Solution

For management and business stakeholders — no jargon, just clarity

🚨 Context & Problem

DGM already monitors the grid.
But operators still interpret it manually.

DGM already provides grid monitoring KPIs and a user experience for viewing Distribution station KPIs. However, operators still need to interpret KPIs manually and translate them into "what happened / why / what next" — especially during incidents and shift handovers.

5 DGM KPIs Currently in Scope
Congestion
Power Factor
Current Unbalance
Voltage Deviation
Infeed Detection
📊

KPIs Exist, Understanding Takes Time

KPIs exist in DGM, but understanding and decision making still takes time. Operators must manually interpret what each value means.

🗣️

No Plain-Language Guidance

Operators want quick, plain-language explanations and guidance — not raw numbers. Current tools don't provide this conversationally.

🧠

Knowledge Hard to Reuse

Knowledge from similar past cases is difficult to reuse quickly — requires searching manuals, notes, and tickets manually.

⚠️
Key Challenge: Keep answers grounded in DGM KPI facts (avoid hallucinations), provide a fast conversational experience without changing how DGM computes KPIs, and add optional knowledge retrieval without mixing it with KPI truth.
✅ The Solution

A conversational assistant
built on top of DGM

DGM remains the "source of truth" for KPI values and KPI logic. LangChain orchestrates tools — one to fetch KPIs from DGM, another to retrieve relevant text context from a Vector DB. GLM-5 routes intent; Qwen narrates and explains in plain language.

🏛️

DGM = Source of Truth

DGM remains the authoritative source for all KPI values and KPI logic. No recomputation in this PoC — facts come directly from DGM.

🔗

LangChain Orchestrates Tools

LangChain builds a tool-calling chain: fetch KPIs from DGM, retrieve context from Vector DB, and optionally generate clarification questions.

💬

GLM-5 Routes · Qwen Explains

GLM-5 handles intent routing and structured extraction. Qwen narrates and explains in plain language that any operator can understand.

💡
Outcome: Operators can ask "Why is this feeder red?" and get: KPI facts (from DGM) + AI explanation + recommended next action. Knowledge retrieval improves speed to resolution, while KPI truth remains governed by DGM.

🔍 Design Choices & Guardrails

Non-negotiable rules.
Grounded. Governed. Reliable.

Every architectural decision ensures KPI truth stays with DGM, retrieved context complements but never overrides facts, and all LLM calls are governed and auditable.

🏛️ Core Rule — Non-Negotiable

  • KPI values come from DGM's existing KPI computation — no recomputation in this PoC
  • 5 KPIs in scope: Congestion, Power Factor, Current Unbalance, Voltage Deviation, Infeed Detection
  • LangChain tool fetches KPI facts; Qwen only narrates what it receives
  • Retrieved text from Vector DB must never overwrite KPI numeric truth
  • Guard checks enforce separation of KPI facts vs. retrieved context

🗄️ Vector DB — Context Only

  • Used ONLY for text retrieval: manual excerpts, SOPs, incident summaries, operator notes
  • NOT used for KPI numeric truth — DGM owns that
  • Retrieval content complements, does not override, DGM KPI facts
  • Response clearly separates "KPI facts" vs "AI explanation" vs "retrieved context"
  • Any vector DB implementation is valid for this PoC
⚡
Reliability Notes: GLM-5 (internally labeled GLM 5.1) has ongoing availability tuning — fallback strategy designed: GLM-5 slow/unavailable → Qwen in strict JSON mode. Recommended default narrator: qwen-3.5-27b (generally available).
🔒 Data Classification ACP-2 only via LLM Gateway. ACP-3 strictly prohibited.
📦 Data Minimization Only pre-computed DGM KPI facts passed to LLMs — never raw telemetry.
📝 Auditability Prompts and responses logged with appropriate redaction.
🚧 Controlled Access Only whitelisted LangChain tool calls permitted.
🔍 Transparency LLM responses explicitly state when data is missing or unavailable.
